# 2026-06-27 Tier 1 x86_64 AVX2 F32 Matvec

## Slice

Tier 1 still needs AVX2 work for Intel and AMD CPU targets. This slice adds the
first x86_64 AVX2 matvec boundary for F32 matrices while preserving the scalar
fallback and the existing aarch64 NEON path.

The implementation:

- adds an `X86_64Avx2` F32 matvec backend behind `target_arch = "x86_64"`;
- dispatches to AVX2 only after `is_x86_feature_detected!("avx2")`;
- keeps the unsafe intrinsic use inside the `scalar/matvec.rs` x86_64 module;
- uses unaligned 8-lane AVX loads plus scalar tail accumulation; and
- makes architecture-specific backend enum variants and tests conditional so
  cross-target checks stay warning-free.

## Validation

Test-first failure before implementation:

```text
cargo check -p ferrite-inference --target x86_64-unknown-linux-gnu --tests
```

The check failed because the x86_64-only test referenced
`F32MatVecBackend::X86_64Avx2`, which did not exist.

Passing checks after implementation:

```text
cargo check -p ferrite-inference --target x86_64-unknown-linux-gnu --tests
cargo test -p ferrite-inference f32_matvec_uses_neon_backend_on_aarch64 -- --nocapture
cargo test -p ferrite-inference --test matvec_kernel_check -- --nocapture
cargo fmt --all -- --check
git diff --check
cargo clippy --workspace --all-targets -- -D warnings
cargo test --workspace
```

The x86_64 target check compiled the AVX2 F32 backend and x86_64-only backend
test without warnings. The local aarch64 NEON F32 backend test passed, the
matvec reference-check target passed all 6 tests, and the full workspace test
suite passed.

## Remaining Work

This is compile-time AVX2 bring-up only. It does not prove AVX2 runtime
correctness on an x86_64 host, does not add AVX2 quantized kernels, and does not
provide Tier 1 throughput evidence. A future x86_64 run must execute the backend
test and reference-check harness on hardware with AVX2 before Tier 1 AVX2
correctness can be marked proven.
